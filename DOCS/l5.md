# Разработка рекомендаций по кодированию (Python)

## 1. Стандарты оформления исходного кода

### Категория: Запреты (Prohibitions)
*Положения, нарушение которых недопустимо ни при каких условиях.*

1.  **Запрещается использовать «магические числа» и строки в коде.**
    *   *Обоснование:* Как и в лекции (запрет числовых констант), жестко заданные значения ухудшают читаемость и усложняют поддержку. В ML это критично для гиперпараметров.
    *   *Пример неправильного использования:*
        ```python
        # Плохо
        if accuracy > 0.95:
            save_model()
        data = load_data('data/train.csv')
        ```
    *   *Пример правильного использования:*
        ```python
        # Хорошо
        ACCURACY_THRESHOLD = 0.95
        TRAIN_DATA_PATH = 'data/train.csv'

        if accuracy > ACCURACY_THRESHOLD:
            save_model()
        data = load_data(TRAIN_DATA_PATH)
        ```

2.  **Запрещается сравнивать логические переменные с `True` или `False` явно.**
    *   *Обоснование:* Аналог запрета из лекции (`== TRUE`). В Python любой объект имеет булево значение, явное сравнение избыточно и может скрыть ошибки (например, если переменная равна 1, а не `True`).
    *   *Пример неправильного использования:*
        ```python
        # Плохо
        if is_trained == True:
            pass
        if has_error == False:
            pass
        ```
    *   *Пример правильного использования:*
        ```python
        # Хорошо
        if is_trained:
            pass
        if not has_error:
            pass
        ```

3.  **Запрещается писать комментарии на языке, отличном от английского.**
    *   *Обоснование:* Прямое требование из лекции. Код может использоваться в международной команде или быть выложен в Open Source.
    *   *Пример неправильного использования:*
        ```python
        # Вычисление функции потерь
        def loss_func(): ...
        ```
    *   *Пример правильного использования:*
        ```python
        # Loss function calculation
        def loss_func(): ...
        ```

### Категория: Требования (Requirements)
*Правила, которые должны выполняться в любом проекте.*

1.  **Все функции и методы должны иметь аннотации типов (Type Hinting) и строки документации (Docstrings).**
    *   *Обоснование:* Аналог требования лекции о типах возвращаемых значений и заголовках функций. В Python это обеспечивает понятность интерфейсов.
    *   *Пример:*
        ```python
        def predict_probability(model,  np.ndarray) -> float:
            """
            Calculates the probability of class 1.

            Args:
                model: Trained sklearn model.
                 Input feature matrix.

            Returns:
                Probability value between 0 and 1.
            """
            return model.predict_proba(data)[0][1]
        ```

2.  **Имена идентификаторов должны следовать соглашению PEP 8.**
    *   *Обоснование:* Аналог требований лекции к именам классов (CamelCase) и функций (lowercase). Единый стиль улучшает читаемость.
    *   *Пример:*
        ```python
        # Классы
        class DataPreprocessor: ...

        # Функции и переменные
        def process_data(): ...
        max_iterations = 100
        ```


3.  **Объявление импортов должно находиться в начале файла, сгруппировано по стандартам.**
    *   *Обоснование:* Аналог требования лекции о размещении глобальных переменных и директив. Упрощает понимание зависимостей.
    *   *Пример:*
        ```python
        # Standard libraries
        import os
        import sys

        # Third-party libraries
        import numpy as np
        import pandas as pd

        # Local applications
        from .utils import load_config
        ```

### Категория: Рекомендации (Recommendations)
*Правила, соблюдение которых облегчает жизнь разработчикам.*

1.  **Рекомендуется ограничивать длину строки 88 символами.**
    *   *Обоснование:* Адаптация требования лекции (100 символов) под стандарт Black/PEP 8. Улучшает чтение кода на разных экранах.
    *   *Пример:*

        ```python
        # Плохо (строка слишком длинная)
        result = very_long_variable_name + another_long_variable_name + yet_another_long_variable_name

        # Хорошо (перенос строки)
        result = (
            very_long_variable_name
            + another_long_variable_name
            + yet_another_long_variable_name
        )
        ```

2.  **Рекомендуется избегать глубокой вложенности циклов и условий (Cyclomatic Complexity).**
    *   *Обоснование:* Развитие идеи лекции об упрощении сопровождения. Если уровень вложенности > 3, стоит вынести логику в отдельную функцию.
    *   *Пример:*
        ```python
        # Рекомендация: вынести внутреннюю логику в функцию check_validity()
        if user:
            if user.is_active:
                if user.has_subscription:
                    # logic here
        ```

3.  **Рекомендуется использовать контекстные менеджеры (`with`) для работы с ресурсами.**
    *   *Обоснование:* Аналог рекомендации лекции о безопасной работе с памятью/переменными. Гарантирует закрытие файлов/соединений.
    *   *Пример:*
        ```python
        # Хорошо
        with open('data.txt', 'r') as f:
            content = f.read()
        # Файл закроется автоматически
        ```

## 2. Чек-лист для проверки кода (Code Review Checklist)

*Перед выставлением кода на инспекцию автор должен самостоятельно проверить следующие пункты:*

1.  [ ] **Стиль именования:** Классы в `CamelCase`, функции и переменные в `snake_case`?
2.  [ ] **Магические числа:** Все ли числовые константы вынесены в переменные с понятными именами?
3.  [ ] **Документация:** Есть ли `docstring` у всех публичных функций и классов?
4.  [ ] **Типизация:** Присутствуют ли аннотации типов (`def func(a: int) -> str:`)?
5.  [ ] **Комментарии:** Все ли комментарии написаны на английском языке?
6.  [ ] **Длина строки:** Не превышает ли длина строки 88 символов?
7.  [ ] **Логика булевых значений:** Отсутствуют ли явные сравнения `== True` / `== False`?
8.  [ ] **Импорты:** Находятся ли импорты в начале файла и отсортированы ли они?
9.  [ ] **Обработка ошибок:** Есть ли обработка исключений (`try-except`) в местах работы с файлами или сетью?
10. [ ] **Чистота кода:** Отсутствуют ли закомментированные блоки кода и неиспользуемые переменные?

## Комментарий наставника (ML Context)

Поскольку ты строишь карьеру в машинном обучении, обрати внимание на следующие нюансы при применении этих стандартов в ML-проектах:

1.  **Конфигурация вместо констант:** В серьезных ML-проектах «вынесенные константы» (как в примере выше) часто заменяют на файлы конфигурации (`.yaml`, `.json` или библиотеки вроде `Hydra`/`OmegaConf`). Это позволяет менять гиперпараметры без изменения кода.
2.  **Воспроизводимость (Reproducibility):** В чек-лист можно добавить пункт о фиксации `random_seed`. Это критично для науки о данных.
3.  **Линтеры:** В реальной работе никто не проверяет чек-лист вручную. Используй инструменты автоматизации: `flake8`, `black`, `pylint`, `mypy`. Они автоматически найдут нарушения большинства пунктов из этого списка.
