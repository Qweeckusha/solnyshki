# Замусориватели: дублирование кода, комментарии

## Дублирование кода

### Как выглядит в коде
```python
def process_user(user):
    if not user.email or not user.name:
        return None
    cleaned_email = user.email.strip().lower()
    cleaned_name = user.name.strip().title()
    return {"email": cleaned_email, "name": cleaned_name}

def process_customer(customer):
    if not customer.email or not customer.name:
        return None
    cleaned_email = customer.email.strip().lower()
    cleaned_name = customer.name.strip().title()
    return {"email": cleaned_email, "name": cleaned_name}
```

### Причины появления
Копирование фрагментов вместо рефакторинга часто возникает из-за сжатых сроков и фокуса на быстрой сдаче фич. Разработчики боятся ломать рабочий код, поэтому предпочитают дублировать логику, а не менять общую реализацию. Отсутствие регулярных код-ревью и парного программирования позволяет дублям закрепляться в кодовой базе. Слабая модульность архитектуры вынуждает писать одинаковые проверки и преобразования в разных модулях. Игнорирование принципа DRY становится привычкой, когда команда не выделяет время на технический долг в спринтах.

### Как исправить?
- Вынести общую логику в отдельные функции или классы.
- Использовать параметризацию и абстракцию вместо copy-paste.
- Рефакторить только под покрытием автотестов.
- Внедрить статический анализ и обязательные код-ревью.

### Выводы. Чем полезно исправление?
Устранение дублей сокращает площадь для багов и ускоряет внесение изменений во всей кодовой базе. Команда тратит меньше времени на синхронизацию правок, что поддерживает устойчивый темп разработки. Это также упрощает поддержку и снижает когнитивную нагрузку при работе с legacy-модулями.

## Комментарии

### Как выглядит в коде
```python
# Устанавливаем возраст в 25
age = 25
# Проверяем, что возраст больше 18
if age > 18:
    # Выводим сообщение о совершеннолетии
    print("Совершеннолетний")
# else не нужен, так как мы не обрабатываем несовершеннолетних
```

### Виды таких комментариев
• Избыточные (дублируют очевидное поведение кода)  
• Закомментированный код (история, которая не нужна в production)  
• Устаревшие (описывают логику, которая уже изменилась)  
• Поясняющие синтаксис или имена переменных  
• Хаотичные TODO без контекста, автора и сроков  

### Как исправить?
- Переименовать сущности в самодокументирующиеся.
- Выносить сложные блоки в функции с говорящими названиями.
- Оставлять комментарии только для пояснения «почему», а не «что».
- Удалять закомментированный код или перенести в отдельный файл.

### Выводы. Чем полезно исправление?
Чистый код читается быстрее, а осмысленные комментарии фокусируются на бизнес-ограничениях и архитектурных решениях. Это снижает когнитивную нагрузку, ускоряет онбординг и делает код-ревью более продуктивным.
